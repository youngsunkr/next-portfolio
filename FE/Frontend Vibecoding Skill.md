# Frontend Architecture & Next.js Generation Protocol

## When to use
Use this skill when creating or modifying Next.js pages, UI components, API fetch utilities, or state structures. This guarantees seamless pairing with the .NET Web API and JWT security flow.

## Quick reference
- **Framework**: Next.js (App Router preferred, using React Server Components where applicable)
- **Language**: TypeScript (Strict typing required)
- **Styling/UI**: Tailwind CSS / Enterprise Design System Component rules
- **Integration**: Axios or Fetch API configured to process JWT bearer tokens natively

## Implementation Procedure (Vibecoding Rules)

1. **File Path Specification**:
   - Every response must begin with the exact target file path (e.g., `// path: src/app/dashboard/page.tsx`).
2. **Full Code Delivery**:
   - Do not optimize for brevity. Output all imports, interface definitions, component layouts, hooks, and export statements in full. No omissions allowed.
3. **API Contracts**:
   - Ensure frontend TypeScript `interface` types exactly match the backend C# DTO classes generated by the backend skill.
4. **JWT & Security Handling**:
   - Implement interceptors to automatically append `Authorization: Bearer <token>` headers to outgoing API requests toward the Gateway.

## Pitfalls & Architectural Rules

- **Client vs. Server Components**: Explicitly declare `'use client'` at the absolute top of the file if the component uses state (`useState`, `useEffect`) or browser-only APIs.
- **Error Boundaries**: Handle API faults gracefully using Next.js `error.tsx` paradigms, mirroring the backend global exception handling philosophy.
- **No Hardcoded Links**: Route definitions must follow Next.js `<Link>` component practices for optimized prefetching.

## Verification Checklist for AI
- Is there any placeholder or hidden logic? (Ensure 100% code completion).
- Are all types explicitly defined without resorting to `any`?
- Is token expiration handled correctly on network requests?
